# AgentWorld 全过程交互提示词总结

## 1. 文档目的

本文档用于汇总本轮 AgentWorld 从立项、设计、实现、重构、纠偏到当前收敛状态的高信号交互提示词。目标不是逐字回放所有对话，而是把真正影响产品方向、技术路线、界面形态、数据治理和扩展机制的要求沉淀为一份可复用的工作底稿。

本文档覆盖：

- 原始目标与首轮约束。
- 过程中对方向的多次纠偏。
- 当前已经稳定下来的硬性要求。
- 后续继续推进时可直接复用的提示词模板。

本文档不覆盖：

- 每一次临时性的命令输出。
- 不再适用的中间实现细节。
- 系统内部的隐藏推理或调试噪音。

## 2. 原始总目标提示词

最初的核心任务可以归纳为一条总提示词：

“请通读代码仓，梳理清楚当前主线方向，去除不必要的设计文档和代码，只保留最核心的部分，并围绕团队级 Agent 平台完成系统定位、分层设计、详细设计、实现与验证，确保不遗漏。”

这条总提示词包含十个一级能力域：

1. Provider 执行层。
2. Agent 定义层。
3. 工具 / Skill 管理层。
4. 多 Agent 编排层。
5. Agent 团队任务执行层。
6. 业务团队管理层。
7. 任务执行展示层。
8. 环境层。
9. 记忆层。
10. 两个可配置案例：代码检视与每日安全检视。

它同时附带三个基础原则：

- 前后端要一起做，不是只做文档。
- 插件化扩展不能污染主干。
- 所有实现都要围绕最终可生产使用的平台，而不是演示页面。

## 3. 第一轮方向修正提示词

在初始目标之后，方向被进一步修正为“先拆任务，再执行”，对应的提示词可以总结为：

“先把指令拆解成可执行措施，进行归并、细化、分层分类和执行计划排布，然后通过多 Agent 方式执行，最后更新 README，不要直接开干。”

这一轮提示词明确了四个方法论要求：

- 先做计划，不直接改页面。
- 关键 case 要基于 AgentWorld 平台能力配置出来，而不是为 case 写死逻辑。
- 外部代码仓、邮件、IM 等差异能力要通过插件开发接入。
- README 必须随着真实能力同步更新。

## 4. 术语与表达风格修正提示词

随后又增加了一条很关键的术语约束：

“之前设计的王国、团队、酒馆等语言风格，只是一种皮肤。先去除这一类语言描述，统一改成严肃标准说法。”

这条提示词的稳定结论是：

- 产品对外表达使用正式治理术语。
- 可以保留语言包和未来换肤能力。
- 默认产品语言不能带叙事化、游戏化命名。

## 5. OpenViking 集成提示词

OpenViking 相关要求是整轮工作中的核心约束之一，可归纳为下面这条提示词：

“基于 OpenViking 做团队级知识管理和记忆层设计，采用真实集成，最好通过二进制方式直接集成进 AgentWorld，启动 AgentWorld 时同步拉起 OpenViking，不依赖 Docker。”

这条提示词稳定出了以下结果：

- 记忆层不能是假的 mock 能力。
- OpenViking 要作为真实服务存在。
- 知识管理要支持团队级、项目级、Agent Team 级分层。
- Skill 存储和知识空间管理要落在 OpenViking 能力之上。
- 部署方式优先本地二进制或发布包内二进制。

## 6. 统一任务蓝图提示词

后续最关键的一轮架构纠偏，是把两个 case 收敛成同一套平台内核。可提炼为：

“不要围绕神盾计划和安全检视分别硬编码，而要抽象出统一的 Task Blueprint 机制，把触发器、输入、执行环境、Agent Team、记忆、输出、看板、重试、并发、归档等能力都收敛到统一模型里。”

这条提示词最终沉淀出以下稳定要求：

- `Task Blueprint` 是唯一任务定义核心。
- 支持 `manual`、`schedule`、`webhook` 三类触发。
- 任务实例化后必须形成 `TaskRun`、`TaskRunNode`、环境快照、权限快照、事件流、Finding。
- 代码检视和每日安全检视都是配置结果，不是主干分支判断。

## 7. Provider 与 Runtime 抽象提示词

Provider 层也经历了一次明确纠偏。总结后的提示词是：

“Provider 执行层不要只是配置几个命令行，要抽象成统一的 Runtime Interface / Adapter 体系；平台负责 control plane，底层运行时负责 session、tool loop、事件流。”

这条提示词对应的稳定目标是：

- 模型执行与平台治理解耦。
- 主干中只保留统一 Runtime 接口。
- 具体引擎通过适配器接入。
- 平台前台不必暴露底层框架名字。
- Provider 配置、Runtime Binding、Model Endpoint 都要作为资源治理。

## 8. Agent 定义层提示词

Agent 层的需求逐步收敛为：

“Agent 是系统第一公民，也是调度的最小单位。Agent 定义必须支持名称、默认系统提示词、角色、权限、工具集、记忆范围、默认模型服务、Harness 能力、可见性、共享团队，并支持 AI 优化定义和真实验证测试。”

稳定要求包括：

- Agent 定义必须是结构化资源，不是自由文本。
- 系统提示词是核心字段。
- 权限、工具、记忆范围、模型服务要能独立配置。
- 测试 Agent 可用性必须是真实调用，不是假按钮。

## 9. Agent Team 编排提示词

多 Agent 团队的高信号提示词可以总结为：

“Team 和业务团队是两个概念。Agent Team 是任务调度单元，要支持 Leader、Worker、并行 / 串行、汇总、冲突处理、共享范围和工作流编排，并且这些内容要通过界面配置进入数据库。”

其稳定要求包括：

- Agent Team 从 Agent Definition 选成员。
- Team 要有工作流定义和编排提示词。
- Team 可见性、可用性、可编辑性要和业务团队权限解耦。
- 交互工作台要能展示 Team 内多 Agent 真实轨迹。

## 10. 团队治理提示词

团队治理相关要求可以收敛为：

“平台定位于团队级 Agent 治理平台，团队是一切工作的重点。基础配置、团队治理、智能体治理、总览这四个域要清晰区分，任务归属团队，团队成员、权限、资产、任务蓝图都要从团队视角治理。”

稳定要求包括：

- 团队是业务归属单位，与人和权限强相关。
- Agent Team 是调度单位，与任务执行强相关。
- 团队治理要覆盖组织结构、成员管理、权限配置、资产治理、任务管理。
- 知识库、Skill、Codebase 等资产都要可分配给团队。

## 11. UI 与交互体验提示词

这部分用户给出的约束非常密集，最终可总结为一条稳定提示词：

“界面要像专业系统控制台，简约、扁平、克制，避免 AI 味、营销味和说明书式页面；用 React 和成熟组件库，采用表格、表单、弹窗、树状结构、可收缩侧边栏、任务会话气泡、实时状态提示和专业后台式布局。”

这条提示词带出了多条具体约束：

- 左侧边栏按四个大块收敛：总览、智能体治理、团队治理、基础配置。
- 菜单字数要短，颜色对比要清晰。
- 表格页优先，新增和编辑优先走弹窗。
- 不要把所有信息平铺在页面上。
- 会话中事件流和摘要不能压过主体对话。
- 多 Agent 对话要像聊天气泡，可观察并行执行状态。
- 输出中要有 thinking / tool use / status animation，但不能把内部隐藏推理当成平台强依赖。

## 12. 插件化提示词

插件化方向的要求可以浓缩为：

“AgentWorld 是开源主系统，代码平台、邮件、IM、Webhook、输出发布都必须通过插件化接入；插件开发不能要求修改主干代码，只能通过扩展点和导入机制接入。”

稳定结果包括：

- 插件要有 manifest。
- 插件至少覆盖 repository connector、webhook parser、output publisher、tool bundle。
- 插件只能通过平台 SDK 访问上下文、环境、密钥引用、权限申请、事件、Finding、artifact。
- 插件不能直接读数据库，不能绕过权限系统。

## 13. CodeHub 特定提示词

CodeHub 相关的高信号提示词是：

“上面给出的 CodeHub skill 不要直接硬编码进主干，要把它拆成插件能力，包括 Repository Connector、Comment Publisher、Webhook Parser、Tool Bundle，并通过插件机制导入，让不同企业代码平台都能按这个方式扩展。”

这条提示词的稳定约束是：

- CodeHub 只能作为插件样例。
- API 能力进插件。
- 检视逻辑和检查规则进 Skill / Knowledge。
- 密钥必须走平台 secret ref，不读取本地私有文件。

## 14. 数据治理与无预置数据提示词

后期又出现了一轮非常关键的治理纠偏：

“所有可配置数据，例如团队、任务等，不能写死在页面代码里，也不能通过简单配置文件配置，要通过数据库入库；不能预置数据，也不能数据库预置，只能从界面配置输入。我没有配置的，你就不要做进去。”

这是当前最硬的一条约束之一，稳定结论如下：

- 不允许 seed 业务数据。
- 不允许通过页面默认值偷偷选中数据库第一条记录。
- 不允许把 case、Provider、团队、任务、Codebase、Skill 预先写进主干。
- 页面可以提供空表和新增入口，但不能替用户生成业务配置。
- 模拟数据只能作为本地运行态测试数据，通过界面或资源 API 写入，不进入代码仓。

## 15. 文案与语言包提示词

文案治理要求总结为：

“所有界面文字都要通过语言包加载，默认提供一套当前语言包，未来可定制或在系统设置中配置；全局排查，避免代码里还有写死文案。”

稳定要求包括：

- 页面、按钮、表头、状态、弹窗、帮助文案都尽量走语言包。
- 新增文案优先进入统一语言包文件。
- 产品术语应支持系统级覆盖。

## 16. 端口与运行方式提示词

最近一轮还有一个明确的运行时约束：

“默认在 7369 端口监听。”

当前稳定结果：

- 默认开发和生产启动端口为 `7369`。
- 仍然保留通过 `PORT` 覆盖的能力。
- OpenViking 默认 CORS 白名单同步到该端口。

## 17. 最终稳定版总提示词

把全过程所有高信号要求收敛后，可以得到一条当前最稳定、最适合继续驱动 AgentWorld 的总提示词：

“在现有 AgentWorld 仓库上，围绕团队级 Agent 治理平台目标继续推进：以业务团队为治理核心，以 Agent 为第一公民和最小调度单位，以 Task Blueprint 为唯一任务定义核心，以 OpenViking 为真实知识与 Skill 底座，以插件体系承载代码仓、Webhook、通知和外部系统接入。所有可配置资源必须通过控制台或资源 API 入库，不允许主干预置业务数据。界面采用专业后台控制台形态，强调表格、表单、弹窗、树状结构、会话轨迹和执行可观测性。代码检视与每日安全检视必须作为平台配置结果打通，而不是为 case 写死主干逻辑。” 

## 18. 可复用提示词模板

### 18.1 平台总控提示词

“请基于当前 AgentWorld 仓库继续推进，不要偏离以下主线：团队级治理、Agent 第一公民、Task Blueprint 唯一任务核心、插件化扩展、OpenViking 真实知识底座、数据库驱动配置、无预置业务数据。”

### 18.2 架构与主线梳理提示词

“请先通读代码仓，梳理当前主线方向，输出保留、合并、删除建议，再开始实施；不要直接开改页面。”

### 18.3 Provider / Runtime 提示词

“请围绕统一 Runtime Interface 推进模型执行层，平台只做 control plane，不要把底层运行框架直接暴露到产品界面。”

### 18.4 Agent 定义提示词

“请完善 Agent 定义层，确保名称、默认系统提示词、角色、模型服务、Harness 权限、工具集、记忆范围、可见性、共享团队和真实验证能力都可配置并落库。”

### 18.5 Agent Team 提示词

“请完善 Agent Team 编排层，确保 Leader、成员、编排提示词、工作流结构、共享范围、查看 / 使用 / 编辑权限都能通过表单配置进入数据库。”

### 18.6 任务蓝图提示词

“请围绕 Task Blueprint 推进任务定义，不要新增第二套任务模型；manual、schedule、webhook 都要落在同一个蓝图结构里。”

## 19. 面向高质量交付的提示词强化

如果目标是让 AgentWorld 在严格交付场景里表现更稳定，不应该把额外的评价逻辑做进产品，而应该把关键要求沉淀成稳定的工程执行规范。可以直接复用下面这组提示词。

### 19.1 代码规范与复杂度控制提示词

“请不要新增演示型页面或额外的展示面板，直接优化仓库质量。优先处理超大文件、职责混杂、重复拼装 JSON、表单默认值分散、权限与配置解析重复等复杂度问题。修改后必须补齐对应的审计脚本或校验入口，让复杂度治理成为持续机制，而不是一次性人工修补。”

这条提示词的目标是：

- 优先减少大文件和高耦合模块的继续膨胀。
- 把复杂度问题显式暴露出来，而不是隐藏在实现里。
- 用 audit 脚本和提交前检查约束代码规范。

### 19.2 AI 工具使用频率与深度提示词

“请把 AI 工具使用能力体现在平台设计、提示词资产和执行轨迹里，而不是写成宣传文案。优化 Agent、Agent Team、Task Blueprint、Skill、Knowledge 的提示词模板，要求任务执行时明确体现：计划分解、工具选择依据、证据引用、结果汇总、失败回退、自检复核、人工介入点。”

这条提示词的稳定要求是：

- 提示词资产要引导 Agent 先规划、再调用工具、再归纳证据。
- 不是只要求‘会调模型’，而是要求‘能形成完整工具链执行闭环’。
- 交互工作台和任务空间的轨迹要能体现工具使用深度。

### 19.3 基础安全检验提示词

“请从基础安全基线出发优化系统：密钥只保留 secret ref；Webhook 必须支持 secret 校验；插件不能读取本地私有 secret 文件；敏感输出必须脱敏；仓库内增加安全审计脚本，检查危险执行、原始 HTML 注入、硬编码密钥、私钥材料和越权 secret 访问路径。”

这条提示词要求：

- 安全基线要沉淀为仓库脚本和实现约束。
- 插件、Webhook、Provider、Connector、Codebase 的 secret 使用要统一收口。
- 敏感信息治理不能靠约定，要靠检查。

### 19.4 交付自检提示词

“每个阶段完成后，不要只看页面是否能打开；必须跑 typecheck、lint、build、i18n audit、config data audit、quality audit、security audit，并把结果反映到 README 或交付文档中。”

这条提示词要求：

- 自检成为交付的一部分。
- 交付说明和工程基线保持同步。
- 质量与安全检查不依赖人工记忆。

## 20. 最小执行模板

后续如果要继续推进高质量交付，可以直接用下面这段提示词作为起点：

“请基于当前 AgentWorld 仓库继续推进，但不要把额外的评价概念做进产品。优先从四个方向优化：第一，代码规范与复杂度控制，减少超大模块继续膨胀并补齐仓库级审计脚本；第二，AI 工具使用深度，把计划分解、工具调用、证据引用、自检复核和人工介入点沉淀进提示词资产与执行轨迹；第三，基础安全检验，统一 secret ref、Webhook secret 校验、敏感输出脱敏和插件权限边界；第四，交付自检，把 typecheck、lint、build、i18n、config-data、quality、security 全部纳入常规检查。修改后同步更新 README 和相关设计文档。” 

### 18.7 插件化提示词

“请把外部系统接入都沉淀成插件扩展点，不允许为特定代码平台、邮件系统、IM 系统或 Webhook 源修改主干业务逻辑。”

### 18.8 CodeHub 提示词

“请把 CodeHub 能力实现为插件样例，拆成 repository connector、webhook parser、comment publisher、tool bundle，通过 manifest 和 SDK 导入，不要写死在主干 case 里。”

### 18.9 数据治理提示词

“请全局排查所有可配置资源，确保不通过 seed、页面默认值、简单配置文件或数据库预置业务数据；我没有配置的，你就不要做进去。”

### 18.10 UI 提示词

“请把界面做成专业系统控制台，少解释、多操作；该用表格的地方用表格，该用表单的地方用表单，新增和编辑尽量走弹窗，不要让页面充满 AI 味和说明书式信息平铺。”

## 19. 后续继续工作的使用建议

如果后续还要继续驱动 AgentWorld，推荐每次新任务都显式带上下面这组最小提示词组合：

1. “先读仓库和现状，再输出 gap list，不要直接动页面。”
2. “保持 Task Blueprint、插件化、OpenViking、数据库驱动配置这四条主线不偏。”
3. “不允许预置业务数据，不允许偷偷选默认第一条记录。”
4. “所有新增能力都要回答三个问题：落在哪个资源模型、如何通过界面配置、如何通过任务蓝图引用。”
5. “界面保持专业后台控制台风格，不要回退成演示壳。”

## 20. 当前结论

本轮全过程的交互提示词，最终已经收敛成一套非常明确的产品和工程约束：

- 平台不是聊天 demo，而是团队级 Agent 治理平台。
- 代码检视和安全检视不是两个系统，而是同一套平台内核的两种配置。
- OpenViking、插件体系、Task Blueprint、数据库驱动配置，是当前主线的四根骨架。
- “不预置业务数据”已经成为当前最重要的交付约束之一。
- 后续所有实现都应该围绕这套稳定提示词继续推进。
