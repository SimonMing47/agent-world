# AgentWorld 核心架构（精简版）

> 本文档替代历史超长设计稿，聚焦“团队级 Agent 平台”主线能力与可扩展落地。

## 1. 系统定位

AgentWorld 定位为 **团队级别的多 Agent 平台**，核心目标：

- 用“业务团队 + Agent 团队 + 任务”作为治理主轴。
- 用“可观测执行过程”替代黑盒式调用。
- 用“插件化扩展”连接外部系统（IM、邮件、代码仓、Provider）。

## 2. 九层架构

1. **Provider 执行层**：统一执行网关，默认 `opencode`，可发现/接入 `claude code`、`openclaw` 等 CLI/SDK。
2. **Agent 定义层**：定义 Agent 的角色、能力、权限、工具集、提示词。
3. **工具/Skill 管理层**：工具和 skill 插件注册、版本、权限、审计。
4. **多 Agent 编排层**：定义团队 Leader、协作 Agent、交互关系、团队目标。
5. **Agent 团队任务执行层**：任务接入、分配、执行空间、完整过程追踪（thinking/tool use/message）。
6. **业务团队管理层**：创建者/编辑者/使用者权限与可见性（团队/全局/个人）。
7. **任务执行展示层**：按团队与任务类别（一次性/定时/webhook）组织看板。
8. **环境层**：代码仓、私钥、执行人、执行路径、沙箱能力。
9. **记忆层**：基于 OpenViking 的分层分域记忆；支持 skill 记忆访问。

## 3. 前后端排布原则

- 前端以“九层”作为导航骨架，而不是按历史名词分散。
- 后端以“领域核心 + 插件边界”组织：
  - `src/server/*-core.ts`：主干能力（调度、执行、追踪、权限、租户）。
  - `src/server/*-adapter.ts`：Provider 或外部系统适配。
  - `src/app/api/**`：按层暴露 API。
- 任务执行页面必须展示结构化时间线：
  - `agent message`
  - `tool use`
  - `tool result`
  - `reasoning summary`
  - `intervention`

## 4. 插件化原则（关键约束）

- 主干仓库保持稳定；外部系统通过插件接入。
- 插件通过能力声明注册（协议、权限、配置 schema、健康检查）。
- IM、邮件、代码仓、Provider 都通过同一插件生命周期管理。
- 新增插件不要求修改主流程代码，只进行扩展挂载。

## 5. 两个可快速扩展的案例

### Case A：神盾计划（MR webhook 代码检视）

1. 配置 webhook 任务模板（全局可见）。
2. webhook 投递 MR diff。
3. 系统路由到已编排的检视 Agent 团队。
4. 团队按记忆层 skill 执行“分层分类检视”（安全/风格/架构/性能）。
5. 结果回写 MR，任务看板实时展示状态与审计轨迹。

### Case B：全量安全检视（定时）

1. 配置“每日定时”任务模板。
2. 定时拉取仓库集合并执行安全扫描链路。
3. 汇总结果进入任务看板与团队视图。
4. 通过邮件插件发送日报。

## 6. 核心可用性与扩展性策略

- 可靠性：任务幂等键、重试策略、租约锁、失败隔离。
- 可用性：降级执行（Provider fallback）、人工接管、回放与重跑。
- 扩展性：
  - Provider/Tool/Channel 都走插件协议；
  - 记忆层支持跨团队分域；
  - 多 Agent 拓扑支持从简单 Leader-Follower 到 DAG 协作。
